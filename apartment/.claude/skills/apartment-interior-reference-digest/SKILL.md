# Apartment Interior Reference Digest

오늘의집·네이버 블로그·업체 포트폴리오 검색 → 점수 평가 → 레퍼런스 요약 파이프라인.
소스별 품질과 광고성 여부가 다양하므로 결과는 의사결정 참고 자료로 취급한다 — 신뢰할 수 있는 시공 견적 피드가 아니다.

## 호출 후 입력 해석

- 검색 범위는 `config/interior-reference-digest.json`과 의사결정 문서에서 읽는다.
- cron 진입이면 마지막 assistant 출력이 Discord 요약으로 쓰일 수 있으므로 짧은 한국어 요약을 반드시 남긴다.
- 사용자가 결정을 확정하기 전에는 새 항목을 검토 후보로만 기록한다.

## 정보 출처 (단일 출처)

- 작업 디렉터리: `~/ai-nodes/apartment`
- 의사결정 노트: `~/ai-nodes/apartment/docs/interior/lucky-5-1004-interior-decisions.md`
- 레퍼런스 노트북: `~/ai-nodes/apartment/docs/interior/interior-references.md`
- 설정 파일: `~/ai-nodes/apartment/config/interior-reference-digest.json`
- 산출물 루트: `~/ai-nodes/apartment/data/interior-reference-digest/`

## 워크플로

### 1단계: 의사결정 문서 + 설정 파악

다음 5개 docs를 읽어 현재 범위·예산·레퍼런스·미결정 항목을 파악한다:

- `lucky-5-1004-interior-decisions.md` (결정 노트 원본)
- `lucky-5-1004-decision-summary.md`
- `lucky-5-1004-field-checklist.md`
- `lucky-5-1004-contractor-brief.md`
- `lucky-5-1004-decision-queue.md` (상태 보드 — 다른 docs에서 신규 미결정/완료 항목이 보이면 먼저 갱신)

이어서 `config/interior-reference-digest.json`을 읽어 검색 키워드·점수 기준·산출물 설정을 확인한다.
`toneReferenceFocus`가 있으면 오늘 추천의 상위 필터로 사용한다. 현재 기본 방향은 화이트/아이보리 기반에 밝은 오크·애쉬 우드를 20~30% 정도만 섞는 톤이다.

### 2단계: 웹 검색

`config.searchQueries` 순서대로 복수의 좁은 쿼리를 실행한다 — 단일 광범위 쿼리 금지.

cron 안정성을 위해 WebSearch는 한 assistant turn에 여러 개 병렬 호출하지 말고 순차 실행한다.
2026-05-27 cron에서 병렬 WebSearch 결과를 받은 뒤 최종 assistant 출력 없이 Claude CLI가 exit 0으로 종료되어 stdout이 비는 문제가 있었다.
각 검색 결과를 받은 뒤 다음 검색으로 넘어가거나 최종 요약을 출력해야 한다.

검색 우선순위(config의 `sourcePriority` 참조):

1. 오늘의집 프로젝트 페이지
2. 네이버 블로그 시공 후기 (지역 업체 중심)
3. 업체 포트폴리오 페이지
4. 브랜드/플랫폼 자료 (보조)

정확한 단지 결과가 부족할 때만 인근·유사 단지로 검색 범위를 넓힌다.

### 3단계: 페이지 수집 및 점수 평가

- 유망한 페이지를 직접 fetch한다.
- `config.scoringRubric`의 루브릭으로 후보 점수를 산정한다.

### 4단계: 리포트 저장

`data/interior-reference-digest/YYYY-MM-DD/report.md`에 날짜별 마크다운 리포트를 저장한다.
한국어 제목은 **"오늘의 인테리어 추천"** 고정 ("다이제스트" 사용 금지).

리포트 형식은 `references/report-template.md` 참조.

### 5단계: 레퍼런스 노트북 갱신

품질 높은 레퍼런스 후보를 `docs/interior/interior-references.md`에 안정적 ID(`R-00X`)로 추가한다.

### 6단계: Discord 요약 출력 (cron 진입 시 필수)

cron은 OpenClaw delivery `announce -> Discord`가 담당한다. Claude skill은 Discord API를 직접 호출하지 않는다.

대신 cron 진입 또는 `run_with_claude.sh` 호출에서는 마지막 assistant 출력(stdout)에 Discord에 그대로 보낼 수 있는 짧은 한국어 요약을 반드시 남긴다.

- `NO_REPLY` 금지
- "완료" 상태만 말하고 끝내기 금지
- 오늘 추천/추가 레퍼런스/오늘 결정할 질문 3개를 포함
- 추천 레퍼런스는 R 번호, 제목, 원문 URL을 함께 표시한다
- 의사결정 질문은 단순 A/B/C만 던지지 말고, 각 질문마다 대표 예시/사진·원문 링크 1~2개와 "처음 인테리어하는 사람이 봐야 할 차이"를 포함한다
- Discord 링크는 임베드 폭주를 줄이기 위해 `<https://...>` 형태로 감싼다
- 마크다운 테이블 금지, 불릿 사용
- 파일 경로는 필요할 때만 마지막 1줄로 짧게 표시

### 7단계: 의사결정 제안 처리

레퍼런스 후보가 기존 결정을 바꿀 가능성이 있을 때:

- 사용자 확인 전까지 "검토 후보"로 레퍼런스 노트북에 기록한다.
- 사용자가 확인한 후 `D-00X` 항목을 결정 노트에 추가한다.

### 8단계: 초보자용 비교 예시 의무화

사용자는 인테리어 경험이 없으므로, 의사결정 질문은 "선택하세요"가 아니라 "비교해서 이해할 수 있게" 작성한다.

각 결정 질문마다 가능한 한 다음을 포함한다:

- 대표 예시 2~3개: 현재 선택지 A/B/C와 연결되는 실제 시공 사례, 제품 페이지, 브랜드/업체 자료
- 사진 확인 가능 링크: 원문 URL 안에 시공 사진/제품 사진이 있는지 확인하고, 가능하면 이미지 URL 또는 사진이 있는 원문 링크를 같이 남긴다
- 차이 설명: 비용, 관리 난이도, 생활감, 하자 리스크, 우리 집 적용성을 초보자 눈높이로 2~3줄 설명
- 추천 이유: "왜 이 선택지가 현재 예산/취향/공사 범위에 맞는지"를 짧게 명시
- 확인 질문: 업체에게 물어볼 실측/견적 질문 1~2개

좋은 질문 예:

- "도배 방식은 A/B/C 중 무엇?"에서 끝내지 말고, 전체 실크/혼합/광폭 합지의 실제 사진 링크와 함께 오염·통기성·비용 차이를 설명한다.
- "조명 색온도 선택"에서 끝내지 말고, 2700K/3000K/4000K가 실제 공간에서 어떻게 보이는지 예시 링크와 함께 설명한다.
- "콘센트 범위 선택"에서 끝내지 말고, TV벽/소파/침대/주방가전 콘센트가 생활에서 왜 필요한지 사례로 설명한다.

## 일일 리포트 형식

리포트는 짧게 유지한다:

- 오늘의 추천 레퍼런스 3~5개, 각 레퍼런스별:
  - 제목/출처
  - 클릭 가능한 원문 URL
  - 한 줄 요약
  - 화이트/우드 톤 평가: 화이트:우드 비율, 우드 색감, 과하지 않은지
  - 우리 집 적용 포인트
  - 업체/글 신뢰도 1차 판단
  - 예산/하자 리스크
  - 비확장 발코니/경계부 문/샷시 보수/욕실/주방/바닥재/수납/작업공간 관련 힌트
- 오늘 결정해볼 질문 3개:
  - 먼저 5개 docs를 훑어 신규 미결정/완료 항목을 확인하고 `lucky-5-1004-decision-queue.md`를 갱신한다
  - 갱신된 "다음 남은 질문"에서 우선순위 높은 항목 3개를 선택한다
  - 최근 7일 리포트에 이미 등장한 같은 주제는 반복하지 않는다
  - "클리어 완료" 또는 결정/방향 확정 항목은 새 근거 없으면 다시 질문으로 올리지 않는다
  - 각 질문은 A/B/C 선택지 + 대표 예시/사진 링크 + 초보자용 비교 포인트 + 짧은 추천 포함
  - 답변 수령 시: `lucky-5-1004-interior-decisions.md`와 `lucky-5-1004-decision-queue.md`에 기록, 필요 시 `decision-summary`, `field-checklist`, `contractor-brief`도 갱신
- 레퍼런스 누적 여부
- 의사결정 문서 반영 후보

Discord 전송 시 마크다운 테이블 금지 — 불릿 사용.

## 경계

**해야 할 것:**

- 확정 결정과 아이디어를 항상 구분한다
- 법적·구조적 주제(발코니 확장 허가, 방화/대피공간, 난방 배관, 외벽/내력벽)는 전문가/관리사무소 확인을 권고한다
- 소스 실패 시 실패 사실을 명시적으로 기록한다
- 같은 날 재실행 호출(예: 9am 정시 cron 외 수동 호출)이 들어와도 새 결정 질문 + 새 추천을 **강제로 생성**한다. 오늘 report.md가 이미 있다는 이유로 skip하지 않는다 (결정 질문 중복 회피는 "최근 7일 다룬 주제" 룰로만 적용 — 같은 날 재실행 자체는 skip 사유 아님)
- 화이트/우드 톤 레퍼런스 요청이 있는 경우, 짙은 월넛·티크가 많거나 우드 루버/템바보드가 과한 카페풍은 우선순위를 낮춘다. 밝은 오크/애쉬 포인트, 웜화이트 벽·천장, 밝은 바닥·가구 조합을 우선한다.

**하지 말아야 할 것:**

- 사용자 승인 없이 업체에 연락하거나 견적을 요청하지 않는다
- 블로그 가격을 최종 비용으로 취급하지 않는다 — "참고용" 표시
- 썸네일만으로 레퍼런스 후보를 채택하지 않는다
- 사용자 확인 없이 `D-00X` 항목을 결정 노트에 직접 추가하지 않는다
- 최근 7일 리포트에서 이미 다룬 주제를 다시 질문으로 반복하지 않는다
- 결정/방향이 확정된 항목을 새 근거 없이 질문으로 다시 올리지 않는다
