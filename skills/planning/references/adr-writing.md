# ADR 작성 원칙

planning 7단계에서 ADR을 작성할 때 적용한다. `planning/SKILL.md` 7단계 수행 전 **반드시** Read 도구로 이 파일을 로드.

ADR은 *왜* 결정했는지를 AI 에이전트가 빠르게 파악하는 단일 출처다. 코드 구현 상세는 ADR이 아니라 코드·`docs/code-architecture.md`·git history가 책임진다.

## 가치 판단 — 이 결정이 정말 ADR감인가

ADR을 쓰기 전에 *이 결정이 정말 ADR감인가* 자문한다. 자명한 결정·단순 refactor·스키마 단일 키 추가는 ADR 아니다. 잘못 ADR로 올리면 토큰 낭비 + 진짜 결정의 가시성을 흐림.

**ADR감 (기록할 것)**:

- 워크스페이스 표준 변경 — 언어 도입/폐기, 디렉터리 패턴 신설, 새 외부 의존성 활성화.
- 둘 이상의 대안 중 *거절 사유*가 있는 선택 (예: Puppeteer 대신 API 직접 호출 — 차단 발견 후).
- 격리 원칙·표준 패턴의 *의도적 비대칭* — 정당화 필수 (예: 다른 워크스페이스 ADR 반대 패턴).
- 외부 시스템 통합 방식 — API endpoint 선택, 인증 흐름, 호출 정책.
- 운영·비용·보안 trade-off — cron 빈도, 캐시 TTL, 비밀 정보 위치.

**ADR감 *아님* (다른 곳에 기록)**:

- DRY 일반 원칙 적용 / 단순 중복 제거 — 자명. git history + 해당 docs로 충분.
- 단일 키 추가·필드 이름 변경 → `data-schema.md` 책임.
- 디렉터리 한 개 추가만 → `code-architecture.md` 책임.
- 흐름 한 단계 변경만 → `flow.md` 책임.
- 단일 버그 수정·이름 변경·import 정리 → git commit 메시지로 충분.
- 일회성 데이터 위치 결정 → 운영 노트 / hand-off 영역.

**경계 판단 규칙**: "이 결정을 *왜* 다른 대안 대신 선택했나"에 *한 문단 이상*의 정당화가 필요하면 ADR. 한 줄로 끝나면 ADR 아님.

## 단일 책임 원칙

한 plan에서 두 결정이 나왔다고 한 ADR에 묶지 않는다. 각 결정은 독립적으로 supersede·인용·추적 가능해야 한다. 묶으면 후속에서 "ADR-N의 어느 부분을 supersede하나"가 모호해진다.

| 같은 ADR로 묶기 OK | 분리 ADR |
|---|---|
| 한 결정이 다른 결정의 *전제 조건* (예: "Naver API 통합 + 쿠키·Bearer 인증 흐름" — 인증은 통합의 일부) | 두 결정이 *독립적으로 supersede 가능* (예: 데이터 단일 출처화 + 기술 언어 도입) |
| 같은 trade-off 축 (예: 캐시 정책 + 캐시 위치) | 다른 trade-off 축 (예: 단일 출처화 + 새 언어 도입) |

**분리할지 모호하면 *분리*가 디폴트**. 통합은 명확히 한 trade-off 축일 때만.

같은 plan 사이클에 ADR 둘 이상이 나오는 건 정상. plan = 실행 단위, ADR = 결정 단위. 일대일 관계 아니다. ADR-A가 ADR-B의 적용 결정을 인용하면 "구현 언어는 ADR-B 참조" 식 cross-link로 처리.

## 구조 표준

각 ADR은 5개 섹션만 가진다:

1. **헤더** — `## ADR-N — 제목` + Status / Date 라인.
2. **맥락** — 왜 이 결정이 필요했는지. 기존 상태 + 발견된 문제. AI 에이전트가 결정의 *왜*를 추론하기에 충분한 만큼만.
3. **결정** — 무엇을 결정했는지. 핵심 trade-off + 거절한 대안 한 줄씩.
4. **결과** — 결정의 의도된 효과 + 알려진 단점.
5. **(선택) 적용** — 어디에 가이드라인이 박혀 있는지 *포인터*만 (예: "`<workspace>/skills/.../runner.sh`의 `attempt()` 함수 패턴 참조"). 코드 블록 금지, 파일 전체 목록 금지.

다음은 ADR에 **넣지 않는다**:

- 코드 블록 (before/after 비교, 함수 구현, 셸 명령 시퀀스).
- 적용 대상 파일의 전수 목록 (디렉터리 한 줄 또는 패턴으로 충분).
- 변경 이력 (git history가 단일 출처).
- 검증 결과의 구체 수치 (`docs/learn/` 또는 task phase 산출물로).
- 미래 enhancement / TODO (별도 plan이나 task로).

코드를 알아야 결정 자체를 이해할 수 없는 경우만 짧은 인용 1-2줄 허용 (예: "기존 `--output-format json` 폐기가 부작용을 일으켰음" 정도).

## 왜 이 원칙인가

- AI 에이전트가 `docs/adr.md`를 읽을 때 소비하는 토큰을 줄인다. 의사결정 맥락에만 집중.
- ADR과 코드 사이 drift 위험 감소 — ADR은 *왜*만, 코드가 *어떻게*의 단일 출처.
- 구현 상세는 `docs/code-architecture.md`(현행 구조) / `docs/learn/`(회고) / git history / task phase 산출물로 자연스럽게 분산.

기존 ADR이 본 원칙에 어긋나면 (예: 코드 블록이 들어가 있음) 그 자리에서 슬림화하지 말고, 별도 `plan{N}-adr-cleanup` 후속 task로 처리한다 — 한 번에 너무 많은 변경을 한 ADR 사이클에 섞지 않는다.
