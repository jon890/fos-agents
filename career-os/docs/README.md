# 문서 작성 가이드 — career-os

career-os 문서를 작성하거나 수정하기 전에 확인하는 문서 책임 가이드다.
문체와 Markdown 형식은 repo root [`../../docs/docs-style.md`](../../docs/docs-style.md)를 따른다.
이 문서는 career-os 안에서 각 문서가 어떤 가치를 제공해야 하는지 정의한다.
또한 각 문서에 무엇을 담지 말아야 하는지 함께 정한다.

## 작성 전 확인

문서 수정 전 아래 순서로 판단한다.

1. 새 내용이 어느 문서의 책임인지 고른다.
2. 같은 내용이 이미 다른 문서에 있는지 확인한다.
3. 현재 상태, 실행 이력, 결정 이유, 구현 상세 중 무엇인지 구분한다.
4. 본문을 추가하기보다 기존 책임 문서로 링크할 수 있는지 먼저 본다.

같은 정의를 여러 문서에 본문으로 복제하지 않는다.
다른 문서가 필요로 하는 정보는 링크와 짧은 요약만 둔다.

## 문서별 책임

### [`prd.md`](prd.md)

- 담을 것: 제품 가치, 사용자 문제, skill 자산 포트폴리오, 다음 가치 방향, 성공 기준.
- 담지 말 것: plan 이력, 구현 세부, 특정 회사·면접 상태, 파일 스키마.

### [`flow.md`](flow.md)

- 담을 것: 입력, 처리 주체, 산출물, 다음 상태, 안전 경계.
- 담지 말 것: 제품 우선순위, JSON schema, 디렉터리 상세, 구현 알고리즘.

### [`data-schema.md`](data-schema.md)

- 담을 것: 파일 위치, JSON 필드, 상태값, 보존 경계.
- 담지 말 것: 제품 가치 설명, 실행 절차, UI 요구사항.

### [`code-architecture.md`](code-architecture.md)

- 담을 것: 디렉터리 책임, skill/script 구조, 외부 연결, 구현 경계.
- 담지 말 것: 제품의 이유, plan 이력, 모든 함수 목록.

### [`adr/`](adr/INDEX.md)

- 담을 것: 왜 결정했는지, 어떤 대안을 왜 버렸는지.
- 담지 말 것: 현재 사용법 전체, 구현 명세, 작업 로그.

### `tasks/plan*/`

- 담을 것: 실행 계획, phase, 검증 기록, 완료 이력.
- 담지 말 것: 영구 제품 계약, 문서별 책임 정의.

## PRD 작성 규칙

PRD는 “무엇을 만들었는가”보다 “어떤 가치를 제공하는가”를 설명한다.
skill은 career-os의 제품 자산이다.
PRD는 각 skill이 사용자 판단과 다음 실행에 어떤 가치를 남기는지 보여줘야 한다.

PRD에 둔다:

- 사용자 문제.
- 제품 약속.
- skill 자산이 제공하는 가치.
- 현재 제품 축.
- 다음에 만들 가치 방향.
- 성공 기준.

PRD에 두지 않는다:

- 완료된 plan 요약.
- 특정 회사명, 면접일, 임시 target.
- 스크립트 내부 구조.
- JSON 필드 목록.
- 폐기된 실행 패턴 설명.

## Flow 작성 규칙

flow 문서는 “무엇이 시작되어 어디까지 이어지는가”를 설명한다.
각 섹션은 아래 질문에 답해야 한다.

- 무엇이 이 흐름을 시작하는가.
- 어떤 입력을 읽는가.
- 누가 처리하는가.
- 어떤 산출물을 쓰는가.
- 다음 상태나 사용자 행동은 무엇인가.
- 어떤 안전 경계에서 멈추는가.

## Data Schema 작성 규칙

data-schema 문서는 저장 구조의 단일 출처다.
새 config, runtime file, report JSON, ledger field를 만들면 여기에 반영한다.

data-schema에는 스키마와 보존 경계를 적는다.
제품 가치, 실행 순서, UI 요구사항은 다른 문서로 보낸다.

## Code Architecture 작성 규칙

code-architecture 문서는 코드와 디렉터리 책임을 설명한다.
새 script, helper, skill directory, 외부 repo 연결이 생기면 여기에 반영한다.

전체 구현 코드나 함수 전수 목록은 쓰지 않는다.
구현 세부는 코드가 단일 출처다.

## ADR 작성 규칙

ADR은 결정의 이유를 남긴다.
코드만 봐서는 알 수 없는 trade-off, 대안 기각, 운영 원칙을 기록한다.

ADR에 두지 않는다:

- 단순 파일 이동.
- 작업 순서.
- 현재 사용법 전체.
- 코드 diff 요약.

## Task 작성 규칙

task는 실행 계약이다.
확정된 문서 결정을 phase로 옮기고, 검증 기준과 보류 조건을 명확히 적는다.

task에 적은 실행 이력은 PRD나 flow 본문으로 승격하지 않는다.
반복될 정책이 되면 이 문서, AGENTS, ADR, skill 중 책임 문서로 흡수한다.

## 공통 금지 패턴

- plan 번호별 완료·계획 목록을 PRD 본문에 누적하지 않는다.
- 특정 회사명, 면접일, 개인 상태를 영구 제품 문서에 박지 않는다.
- 같은 표나 설명을 PRD와 flow에 동시에 본문으로 두지 않는다.
- 폐기된 runner, dispatcher, CLI 패턴을 현재 사용법처럼 설명하지 않는다.
- 문서가 너무 길어지면 새 섹션을 추가하기 전에 다른 문서의 책임인지 다시 판단한다.
