# fos-study 작성 규칙 레퍼런스

이 문서는 `fos-study/CLAUDE.md`와 fos-study 내부 Claude skill들을 바탕으로, study-pack 생성 시 반드시 참고해야 할 문서 작성 규칙을 추린 것이다.

## 1. 저장소 역할
- `fos-study`는 개인 기술 블로그 겸 학습 기록 저장소다.
- 빌드/테스트보다 파일 작성과 구조 관리가 핵심이다.
- 최종 출력은 항상 게시 가능한 markdown 품질을 목표로 한다.

## 2. 폴더 배치 기준
- `task/` — 실제 업무 구현 경험, 특정 시점의 사례 기록
- `architecture/`, `database/`, `java/` 등 — 개념 문서, 기술 설명
- 사례는 `task/`, 일반 개념은 기술 폴더에 둔다.
- Kafka는 `database/`가 아니라 최상위 `kafka/` 유지

## 3. 문서 유형 분리
- 개념 문서는 언제 읽어도 유효한 설명을 지향한다.
- 업무 기록 문서는 진행 기간, 문제 맥락, 구현 배경을 포함한다.
- 개념 문서에서 사례를 언급할 수는 있지만, 사례 본문 전체는 `task/`에 두는 것이 기본이다.

## 4. 민감 정보 제거
업무 경험 문서/사례 매핑 시 다음을 조심한다.
- 내부 URL, IP, 시스템 식별자는 제거 또는 일반화
- 비즈니스 도메인 고유 클래스명, 상품명, 파트너 명칭은 일반화 검토
- 오픈소스/공개 기술명은 그대로 유지 가능

## 5. 문체
- AI 요약체보다 직접 공부한 사람이 정리한 톤을 선호한다.
- 과한 '정리하면', '종합하면', '매우 중요하다' 같은 메타 서술은 줄인다.
- 실제 판단과 trade-off, 왜 그렇게 했는지를 우선한다.
- 개인 회고는 가능하지만 과장된 자기 PR 톤은 피한다.

## 6. 코드/예시 인용
- 실제 코드 흐름을 반영하되, 필요한 경우 구조 중심으로 추상화한다.
- 민감한 회사 내부 구현은 구조와 아이디어 중심으로 일반화한다.
- 인터뷰용 study-pack은 bad vs improved 예제, 실전 실패 사례, trade-off 설명을 선호한다.
- 모든 fenced code block은 여는 fence에 언어를 명시한다. Java/Spring 예시는 ```java, SQL은 ```sql, shell 명령은 ```bash, YAML은 ```yaml, JSON은 ```json, 로그/출력은 ```text를 사용한다.
- bare triple backticks(``` 단독)로 코드블록을 열지 않는다.

## 7. 링크와 구조 연결
- 관련 상세 문서가 이미 있으면 상대 경로 링크를 우선한다.
- 무조건 새 문서를 만들기보다, 기존 문서 업데이트 / 사례 매핑 / 브리지 문서를 먼저 검토한다.

## 8. study-pack 생성 시 적용 원칙
- create/update/augment/bridge 중 하나를 선택한다.
- 기존 문서와의 중복을 줄인다.
- 개념 문서라면 사례 문서를 연결하고, 사례 문서라면 일반 개념 문서로 역링크할 수 있게 구성한다.
- 최종 문서는 면접 대비와 실제 학습 둘 다 가능해야 한다.

## 9. Markdown 렌더링 안전장치

study-pack은 blog-post-writer와 같은 fos-blog 렌더러를 공유하므로, blog-post-writer `references/markdown-pitfalls.md`와 동일한 렌더링 안전 기준을 따른다.

### 9-1. 작성 직후 통합 자가점검 — 스크립트 점검

글을 저장한 직후 아래 grep을 순서대로 실행한다.
하나도 건너뛰지 않는다.

```bash
FILE=<작성한 글 경로>

grep -nE '\*\*"|"\*\*' "$FILE"      # L. bold 안 따옴표 → 따옴표 제거
grep -nE '[┌│▼└┐┘─├]' "$FILE"      # M. ASCII 박스 → mermaid 전환 (트리 제외)
grep -n "~" "$FILE"                  # G. paragraph당 ~ 2개+ → 이스케이프
grep -n "§" "$FILE"                  # H. § 금지 → 평문으로 치환
grep -nE "^#+ [0-9]+\." "$FILE"      # K. heading 숫자 prefix 금지 (fos-blog 자동 번호와 이중 충돌)
grep -nE "(위 [0-9]+|[0-9]+개 항목|[0-9]+번 항목|섹션 [0-9]+)" "$FILE"  # K. 본문 숫자 cross-ref
```

위 중 하나라도 결과가 나오면 해당 줄을 수정한 뒤 재검사한다.

### 9-2. 수동 점검 (스크립트로 잡히지 않는 것)

- **I. 영문 용어 직역 의심** — 한글이 더 익숙한 용어가 영문 그대로 있으면 `WebSearch`로 짧게 확인 후 한글 우선 + 영문 병기.
  의심 가는 것만 확인한다.
- **J. 콤마 3개+ 나열 → 리스트 변환** — 한 단락에 항목이 3개 이상 콤마로 연결되어 있으면 bullet list로 분리한다.
- **명사형 종결 → 동사 종결** — 본문 평문 문장이 "필요.", "불변.", "확보." 같이 끝나면 동사로 풀어 쓴다.
- **A. 인사이트 blockquote 2\~3개 이하** — `> **인사이트**` 블록이 이 개수를 넘으면 정말 필요한 것만 남기고 나머지는 본문에 녹인다.
- **cross-link 대상 파일 존재 확인** (`ls`) + **H1 제목을 표시 텍스트로** 사용했는지 점검.

### 9-3. `~` 취소선 함정

GitHub Flavored Markdown은 한 paragraph 안에 `~`가 두 번 이상 등장하면 두 `~` 사이를 취소선으로 렌더링할 수 있다.
범위 표기와 기간 표기를 자주 쓰는 기술 문서에서 특히 조심한다.

위험한 예:

- `60~100건, 30분~1시간`
- `2026.04 ~ 2026.05 동안 1~2분 지연`
- `10~20ms ... 30~40ms`

대응 원칙:

- 범위 표기의 `~`는 기본적으로 백슬래시 이스케이프한다: `60\~100건`, `1\~2분`
- 기간은 가능하면 en dash를 쓴다: `2026.04–2026.05`
- 코드 인라인/코드 블록 안의 `~`는 그대로 둔다.
- 글 작성 후 `grep -n "~" <file>`로 남은 물결표를 확인한다.

### 9-4. 링크 표시 텍스트 — H1 제목 사용, 백틱 경로 금지

`[\`path.md\`](path.md)` 형태는 fos-blog 렌더링 시 백틱과 경로가 그대로 노출된다.
**반드시 대상 문서의 H1 제목을 표시 텍스트로** 사용한다.

```markdown
# ❌ 잘못 — 경로가 본문에 노출
관련 개념은 [`connection-pool.md`](./connection-pool.md) 참고.

# ✅ 올바름 — H1 제목으로
관련 개념은 [커넥션 풀 크기](./connection-pool.md) 참고.
```

H1 추출:

```bash
grep -m1 '^# ' <대상 파일 경로> | sed 's/^# //'
```

링크 걸기 전에 해당 파일이 실제로 존재하는지 확인한다.
없는 파일에 링크를 걸지 않는다.

### 9-5. bold+괄호 분리 — 이탤릭+괄호 금지

`*한글(영문)*` 같은 이탤릭+괄호 조합은 GFM/fos-blog 파서가 단어 경계 인식을 못 해 강조가 걸리지 않는다.
**bold + 괄호 분리** 형태로 작성한다.

```markdown
# ❌ 잘못 — 이탤릭이 안 걸림
*격벽(bulkhead)*을 세우는가

# ✅ 올바름
**격벽**(bulkhead)을 세우는가
```

### 9-6. cross-link 후보 발굴 (작성 직전 실행)

글의 핵심 키워드로 fos-study 전역을 ripgrep 해 링크 후보를 발굴한다.

```bash
KEYWORDS=("Outbox" "HikariCP" "Circuit Breaker")   # 글의 H1·H2 기반 5~10개 선정
FOS_STUDY_ROOT=<로컬 fos-study clone 경로>

for kw in "${KEYWORDS[@]}"; do
  echo "== $kw =="
  rg -l --type md "$kw" "$FOS_STUDY_ROOT" \
    | grep -v "/node_modules/" | grep -v "$(basename <현재 글 경로>)" | head -5
done
```

발굴된 후보 파일의 H1을 추출하고, 본문 흐름상 자연스러운 자리 1\~2건만 링크한다.
한 paragraph에 링크 3개를 넘기면 가독성이 떨어지므로 가장 가까운 것만 박는다.

### 9-7. 학습 문서 특유 AI-slop 안티패턴

다음 패턴이 보이면 수정한다.

| 패턴 | 문제 | 대응 |
|---|---|---|
| `> **인사이트**` 블록이 4개 이상 | AI가 배치한 하이라이트처럼 읽힌다 | 2\~3개 이하로 줄이고 나머지는 본문에 녹인다 |
| 같은 인사이트가 여러 글에 반복 | 복사-붙여넣기 느낌 | 허브 글 하나를 지정하고 나머지는 링크로 처리한다 |
| 영문 용어 직역 | 한국 독자가 멈추게 된다 | 한글 우선 + 첫 등장 시 영문 병기 (예: "공통 응답 포맷(response envelope)") |
| 콤마로 4개+ 항목 연결 | 눈이 흐른다 | bullet list로 분리한다 |

> 이 규칙은 blog-post-writer `references/markdown-pitfalls.md`와 같은 fos-blog 렌더러를 공유하므로 내용을 맞춘 것이다.
